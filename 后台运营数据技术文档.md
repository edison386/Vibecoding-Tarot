# 后台运营数据技术文档

本文档用于说明 TAROT 项目新增的运营后台数据能力，包括：

- 数据采集链路
- SQLite 存储结构
- 后台接口与鉴权方式
- 前端后台页的数据消费方式
- 部署配置与验证方法

适用对象：

## 2. 实现总览

当前运营后台基于现有单进程 Node.js 服务实现，没有引入独立后台应用。

核心实现文件：

- 服务入口：[src/server.js](/Users/baichuan/Documents/VibeCoding/TAROT/src/server.js)
- 数据存储与聚合：[src/lib/analyticsStore.js](/Users/baichuan/Documents/VibeCoding/TAROT/src/lib/analyticsStore.js)
- 后台页面：[public/admin.html](/Users/baichuan/Documents/VibeCoding/TAROT/public/admin.html)
- 后台前端逻辑：[public/admin.js](/Users/baichuan/Documents/VibeCoding/TAROT/public/admin.js)
- 公共埋点与 visitor 管理：[public/shared.js](/Users/baichuan/Documents/VibeCoding/TAROT/public/shared.js)

整体结构如下：

```text
浏览器
  -> public/shared.js 生成 visitor_id 并上报 pageview
  -> /api/reading/*   在提问 / 抽牌 / 解读过程中写入埋点
  -> src/server.js
  -> src/lib/analyticsStore.js
  -> SQLite: data/tarot.db
  -> /api/admin/overview
  -> /api/admin/readings
  -> public/admin.html + public/admin.js 渲染后台
```

## 3. 数据采集链路

### 3.1 visitor_id

前端在 [public/shared.js](/Users/baichuan/Documents/VibeCoding/TAROT/public/shared.js) 中使用 `localStorage` 持久化访客标识：

- key：`tarot_visitor_id`
- 发送方式：所有请求默认附带 `X-Visitor-Id`

如果浏览器无法读取 `localStorage` 或请求头缺失，后端会按匿名访客处理，不阻断主流程。

### 3.2 页面访问埋点

前端通过：

```http
POST /api/analytics/pageview
```

上报页面访问，当前只识别以下页面：

- `/` 或 `/index.html` -> `landing_view`
- `/draw.html` -> `draw_view`
- `/reading.html` -> `reading_view`

请求体字段：

```json
{
  "path": "/",
  "referrer": "https://example.com",
  "utm_source": "newsletter",
  "utm_medium": "email",
  "utm_campaign": "launch",
  "session_id": "optional"
}
```

### 3.3 占卜主流程埋点

现有公共 API 保持兼容，同时在服务端写入运营数据：

#### `POST /api/reading/start`

行为：

- 创建内存会话
- 写入 `reading_sessions`
- 写入事件 `reading_started`

附带信息：

- 问题原文
- 问题长度
- 来源 referrer / UTM
- landing_path
- device_type
- user_agent

#### `POST /api/reading/:sessionId/select`

行为：

- 更新 `selected_count`
- 首次选牌时写入 `first_select_at`
- 第三张牌完成时写入 `completed_select_at`
- 每次选牌写入 `card_selected`
- 第三张牌额外写入 `selection_completed`

#### `POST /api/reading/:sessionId/reveal`

行为：

- 开始解读时写入 `reveal_started`
- 成功时更新 `reveal_status = success`
- 失败时更新 `reveal_status = error`
- 记录 `analysis_source`
- 记录 `ai_status`
- 记录 `ai_latency_ms`
- AI 回退时额外写入 `ai_fallback`

AI 统计状态规则固定为：

- `disabled`：未启用 AI
- `success`：启用 AI 且本次返回 `spark`
- `fallback`：启用 AI 但本次回退模板

## 4. SQLite 存储设计

### 4.1 数据库位置

默认数据库路径：

```text
./data/tarot.db
```

可通过环境变量覆盖：

```bash
SQLITE_PATH=/opt/tarot-app/data/tarot.db
```

### 4.2 SQLite 初始化策略

在 [src/lib/analyticsStore.js](/Users/baichuan/Documents/VibeCoding/TAROT/src/lib/analyticsStore.js) 中，应用启动时会：

1. 自动创建数据库目录
2. 打开 SQLite
3. 设置 `WAL`
4. 自动建表
5. 自动建索引

关键 pragma：

- `journal_mode = WAL`
- `foreign_keys = ON`

如果 SQLite 初始化失败：

- 主站占卜功能继续可用
- 后台接口返回 `503 analytics unavailable`
- stderr 会输出 analytics 初始化错误

### 4.3 表结构

#### `reading_sessions`

用于存储“一次占卜会话”的聚合状态。

| 字段 | 类型 | 说明 |
|------|------|------|
| `session_id` | TEXT PK | 占卜会话 ID |
| `visitor_id` | TEXT | 访客 ID |
| `question_text` | TEXT | 用户问题原文 |
| `question_length` | INTEGER | 问题长度 |
| `created_at` | INTEGER | 会话创建时间戳 |
| `first_select_at` | INTEGER NULL | 首次选牌时间 |
| `completed_select_at` | INTEGER NULL | 选满三张时间 |
| `reveal_started_at` | INTEGER NULL | 开始解读时间 |
| `reveal_completed_at` | INTEGER NULL | 解读完成时间 |
| `reveal_status` | TEXT | `pending/success/error` |
| `selected_count` | INTEGER | 当前已选牌数量 |
| `analysis_source` | TEXT NULL | `spark/template` |
| `ai_status` | TEXT NULL | `success/fallback/disabled` |
| `ai_latency_ms` | INTEGER NULL | AI 解读耗时 |
| `referrer` | TEXT NULL | 来源引用页 |
| `utm_source` | TEXT NULL | UTM Source |
| `utm_medium` | TEXT NULL | UTM Medium |
| `utm_campaign` | TEXT NULL | UTM Campaign |
| `landing_path` | TEXT NULL | 落地路径 |
| `device_type` | TEXT NULL | `mobile/desktop/tablet/unknown` |
| `user_agent` | TEXT NULL | 原始 UA |

#### `reading_events`

用于存储事件流。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER PK AUTOINCREMENT | 事件自增 ID |
| `session_id` | TEXT NULL | 会话 ID |
| `visitor_id` | TEXT NULL | 访客 ID |
| `event_type` | TEXT | 事件类型 |
| `event_at` | INTEGER | 事件时间戳 |
| `path` | TEXT NULL | 所属路径 |
| `payload_json` | TEXT NULL | 扩展 JSON 数据 |

当前使用的事件类型：

- `landing_view`
- `draw_view`
- `reading_view`
- `reading_started`
- `card_selected`
- `selection_completed`
- `reveal_started`
- `reveal_succeeded`
- `reveal_failed`
- `ai_fallback`

### 4.4 索引

当前索引：

- `idx_reading_sessions_created_at`
- `idx_reading_sessions_reveal_completed_at`
- `idx_reading_events_event_at`
- `idx_reading_events_type_event_at`
- `idx_reading_events_session_id`

这些索引主要服务于：

- 时间范围筛选
- 漏斗统计
- 趋势聚合
- 最近会话查询

## 5. 后台聚合逻辑

### 5.1 时间范围

后台支持的时间范围：

- `1d`
- `7d`
- `30d`

非法值会回退到：

```text
7d
```

### 5.2 概览统计

`getAdminOverview()` 返回以下结构：

#### `summary`

| 字段 | 说明 |
|------|------|
| `landing_views` | 首页访问数 |
| `reading_starts` | 发起占卜数 |
| `selection_completed` | 选满三张数 |
| `reveal_successes` | 解读成功数 |
| `completion_rate` | `selection_completed / reading_starts` |
| `reveal_rate` | `reveal_successes / reading_starts` |

#### `funnel`

每个阶段返回：

```json
{
  "count": 123,
  "conversion_rate": 0.4567
}
```

其中：

- `landing_views.conversion_rate = 1`
- `reading_starts.conversion_rate = reading_starts / landing_views`
- `selection_completed.conversion_rate = selection_completed / reading_starts`
- `reveal_successes.conversion_rate = reveal_successes / selection_completed`

#### `ai`

| 字段 | 说明 |
|------|------|
| `success_count` | AI 成功次数 |
| `fallback_count` | AI 回退次数 |
| `disabled_count` | 未启用 AI 次数 |
| `success_rate` | `success / (success + fallback)` |
| `avg_latency_ms` | AI 平均耗时 |
| `p95_latency_ms` | AI P95 耗时 |

注意：

- `disabled` 状态不会计入耗时统计
- `success_rate` 只基于启用了 AI 的请求计算

#### `traffic`

包含：

- `top_referrers`
- `top_utm_campaigns`
- `device_breakdown`

均返回：

```json
[
  { "referrer": "https://example.com", "count": 12 }
]
```

或对应字段名版本。

#### `trend`

趋势只统计三类事件：

- `landing_view`
- `reading_started`
- `reveal_succeeded`

聚合维度：

- `1d`：按小时 bucket
- `7d` / `30d`：按天 bucket

返回结构：

```json
[
  {
    "bucket": "04-06 18:00",
    "landing_views": 12,
    "reading_starts": 6,
    "reveal_successes": 4
  }
]
```

### 5.3 最近占卜记录

`getRecentReadings()` 返回字段：

- `session_id`
- `created_at`
- `question_text`
- `selected_count`
- `reveal_status`
- `analysis_source`
- `ai_status`
- `ai_latency_ms`
- `device_type`
- `referrer`

默认参数：

- `range = 7d`
- `limit = 50`
- `offset = 0`

限制规则：

- `limit` 最大 `200`
- 非法 `offset` 回退到 `0`

## 6. 后台接口说明

### 6.1 鉴权方式

后台接口不使用 cookie / session。

鉴权方式：

- 浏览器在 [public/admin.js](/Users/baichuan/Documents/VibeCoding/TAROT/public/admin.js) 中把管理员输入的 Token 保存在 `sessionStorage`
- 每次请求附带：

```http
Authorization: Bearer <ADMIN_TOKEN>
```

服务端在 [src/server.js](/Users/baichuan/Documents/VibeCoding/TAROT/src/server.js) 中使用 `crypto.timingSafeEqual()` 做常量时间比较。

鉴权失败响应：

```json
{ "error": "unauthorized" }
```

状态码：

```text
401
```

如果未配置 `ADMIN_TOKEN`：

```json
{ "error": "admin api disabled" }
```

状态码：

```text
503
```

### 6.2 `GET /api/admin/overview`

请求：

```http
GET /api/admin/overview?range=7d
Authorization: Bearer <ADMIN_TOKEN>
```

返回：

- `range`
- `generated_at`
- `summary`
- `funnel`
- `ai`
- `traffic`
- `trend`

### 6.3 `GET /api/admin/readings`

请求：

```http
GET /api/admin/readings?range=7d&limit=50&offset=0
Authorization: Bearer <ADMIN_TOKEN>
```

返回：

```json
{
  "range": "7d",
  "limit": 50,
  "offset": 0,
  "readings": []
}
```

### 6.4 `POST /api/analytics/pageview`

这是运营埋点接口，不是后台管理接口。

返回：

```json
{
  "ok": true,
  "tracked": true
}
```

如果当前 `path` 不在白名单中，`tracked` 会是 `false`。

## 9. 部署与验证

### 9.1 部署要点

上线后台前至少确认：

1. 代码已同步到服务器
2. `npm install` 已执行
3. `better-sqlite3` 已安装成功
4. `ADMIN_TOKEN` 已写入 `systemd`
5. 数据目录存在

示例：

```bash
mkdir -p /opt/tarot-app/data
```

### 9.2 启动后验证

```bash
curl http://127.0.0.1:3000/healthz
curl -I http://127.0.0.1:3000/admin
curl -H "Authorization: Bearer <ADMIN_TOKEN>" "http://127.0.0.1:3000/api/admin/overview?range=7d"
```

可进一步检查数据库文件：

```bash
ls -l /opt/tarot-app/data
```

正常情况下应出现：

```text
tarot.db
```

## 10. 异常与排障

### 10.1 `/admin` 返回 404

优先检查：

- `public/admin.html` 是否已部署
- `public/admin.js` 是否已部署
- `src/server.js` 是否包含 `/admin -> /admin.html` 映射
- `systemctl restart tarot-app` 是否已执行成功

### 10.2 后台页能打开但没有数据

优先检查：

- `ADMIN_TOKEN` 是否配置
- 前端输入的 Token 是否正确
- `/api/admin/overview` 是否返回 `401` 或 `503`
- SQLite 初始化是否失败

查看日志：

```bash
journalctl -u tarot-app -n 100 --no-pager
```

### 10.3 `analytics unavailable`

说明 SQLite 层不可用，常见原因：

- `SQLITE_PATH` 指向不可写目录
- 数据目录不存在
- 依赖 `better-sqlite3` 安装失败

### 10.4 手动运行 `node src/server.js` 报 `EADDRINUSE`

这通常不是故障，而是 `systemd` 已经启动了同一个服务，3000 端口已被占用。

不要在服务已运行时再手动起第二个 Node 进程。

### 10.5 数据为空但页面正常

说明后台功能正常，只是当前时间范围内尚未产生埋点。可以通过访问首页、发起占卜、走完整个抽牌和解读流程来生成数据。

## 11. 当前限制

当前实现仍有这些限制：

- 会话主数据仍保存在内存 `Map`，不是持久化会话系统
- 运营后台数据仅做单机 SQLite，不支持多实例共享
- 未做角色权限、多管理员、审计日志
- 未做问题文本脱敏，当前是完整保存
- pageview 只对白名单路径写入事件
- 没有接第三方 BI / 数据仓库
