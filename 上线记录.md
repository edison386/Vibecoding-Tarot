## 本次上线过程

### 1. 初始方案评估

最开始服务器是 Windows Server。由于本项目是典型的 Node.js 小型 Web 服务，Windows 上部署存在以下问题：

- `node` 未安装
- `winget` 不可用
- 文件上传和服务托管较繁琐
- 后续反向代理和 HTTPS 配置相对麻烦

结论：改用 Linux 更适合本项目。

### 2. 操作系统切换

服务器最终重装为：

- `Ubuntu 24.04 LTS`

选择原因：

- Node / Caddy / Nginx / Docker 生态资料最全
- 部署链路简单
- 后续维护成本低

### 3. SSH 与网络排查

初期从本地上传项目时，出现过以下问题：

- 错用了公网 IP：最初使用的是 `42.21.252.175`
- 正确公网 IP 实际为：`42.121.252.175`
- 服务器内 SSH 服务正常
- `ufw` 已放行 `22/80/443`

最终确认：

- `ssh.service` 正常运行
- `22` 端口监听正常
- 网络超时的根因是公网 IP 填错

### 4. 上传项目到服务器

由于阿里云 Workbench 辅助账号 `ecs-assist-user` 不适合直接用于 `scp`，后来新建了一个普通用户：

- 用户名：`deploy`

本地通过 `scp` 上传项目到服务器用户目录，再移动到正式路径：

- 最终项目目录：`/opt/tarot-app`

项目目录内容确认正常：

- `Dockerfile`
- `package.json`
- `public`
- `README.md`
- `src`
- `tests`

### 5. Docker 方案尝试与放弃

曾尝试使用 Docker 部署，但构建时出现：

- 拉取 `docker.io/library/node:20-alpine` 超时

根因：

- 服务器访问 Docker Hub 超时

结论：

- 暂时放弃 Docker 方案
- 改为直接在 Ubuntu 上安装 Node.js 运行服务

### 6. 安装 Node.js

直接使用 Node 官方 Linux 二进制包安装：

- 版本：`v20.20.1`

安装后验证：

- `node -v`
- `npm -v`

满足项目对 Node `>=20` 的要求。

### 7. systemd 托管 Node 服务

创建了 `systemd` 服务文件：

- 路径：`/etc/systemd/system/tarot-app.service`

核心配置：

- 工作目录：`/opt/tarot-app`
- 启动命令：`/usr/local/bin/node /opt/tarot-app/src/server.js`
- 服务名：`tarot-app`
- 自动重启：开启

启动与验证：

- `sudo systemctl daemon-reload`
- `sudo systemctl enable --now tarot-app`
- `sudo systemctl status tarot-app`

验证结果：

- 服务正常运行
- 本地监听地址：`127.0.0.1:3000`
- 健康检查接口：`/healthz`

### 8. Caddy 反向代理

安装了 Caddy 作为反向代理，并将外部请求转发到：

- `127.0.0.1:3000`

早期问题：

- Caddy 默认欢迎页仍在生效

原因：

- `Caddyfile` 仍是默认配置

处理过程分两个阶段：

- 第一阶段：使用公网 IP 作为临时访问入口
- 第二阶段：域名 `lovetarot.fun` 解析生效后切换到正式域名，并开启 HTTPS
- `www` 自动跳转到主域名

当前正式生效的 `Caddyfile` 结构：

```caddyfile
www.lovetarot.fun {
    redir https://lovetarot.fun{uri} permanent
}

lovetarot.fun {
    reverse_proxy 127.0.0.1:3000
}
```

### 9. AI 功能接入

项目已接入讯飞星火 AI，配置方式为 `systemd` 环境变量注入。

关键环境变量：

- `SPARK_API_PASSWORD`
- `SPARK_MODEL`
- `AI_READING_TIMEOUT_MS`

在 `tarot-app.service` 中补充后，重载并重启服务：

- `sudo systemctl daemon-reload`
- `sudo systemctl restart tarot-app`

验证结果：

- 接口返回 `analysis_source: "spark"`
- 说明当前解析结果已成功走星火 AI，而不是本地模板回退

## 本次遇到的关键问题

### 1. Windows 部署成本高

表现：

- 缺少 Node
- 缺少 `winget`
- 上传和托管不顺手

处理：

- 更换为 Ubuntu 24.04 LTS

### 2. 公网 IP 使用错误

表现：

- `scp` / `ssh` 连接超时

处理：

- 修正为 `42.121.252.175`

### 3. Workbench 辅助账号不适合 `scp`

表现：

- `scp: Received message too long`

处理：

- 新建普通用户 `deploy`

### 4. Docker Hub 超时

表现：

- 无法拉取 `node:20-alpine`

处理：

- 放弃 Docker
- 改为 Node 直接运行

### 5. Caddy 默认欢迎页覆盖业务站点

表现：

- 外部访问出现 `Caddy works!`

处理：

- 重写 `/etc/caddy/Caddyfile`

### 6. 域名初期未生效

表现：

- `dig lovetarot.fun +short` 无结果
- `curl` 提示无法解析域名

原因：

- 域名实名认证未完成
- 域名一度处于 `serverHold`

结论：

- 上线初期只能先用公网 IP 访问
- 域名审核与解析恢复后，已切换到正式域名访问

### 7. 域名切换与 HTTPS 签发

处理过程：

- 在阿里云 DNS 中添加 `@` 和 `www` 的 A 记录，均指向 `42.121.252.175`
- 验证 `dns29.hichina.com` / `dns30.hichina.com` 权威解析已返回正确 IP
- 重载 Caddy
- 等待 Caddy 自动签发并加载证书

验证结果：

- `dig lovetarot.fun +short` 返回 `42.121.252.175`
- `curl -I http://lovetarot.fun` 返回 `308 Permanent Redirect`
- `curl -I https://lovetarot.fun` 返回 `HTTP/2 200`
- 说明域名和 HTTPS 已正式生效
